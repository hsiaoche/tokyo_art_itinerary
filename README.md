# 點線極簡：多線分流同步計畫 (Minimalist Sync Plan)

這是一個專為「多人自助旅遊、多線並行」場景打造的極簡風格靜態網頁應用程式 (Static Web App)。

傳統的旅遊行程表多為單線結構，難以應付多人出遊時「白天各自探索、晚上集合會合」的分流需求。本專案透過「點與線」的極簡視覺語彙，結合時間軸與多軌切換系統，讓所有成員能在同一個介面下掌握全局，隨時切換查看不同路線的當前進度。

---

## / 核心體驗與 UX 策略 (UX Strategy)

在介面與互動設計上，本專案刻意屏棄了多餘的裝飾與原生的手勢盲點，專注於資訊的純粹性與操作的穩定性：

*   **無表情符號的純粹排版 (Typographic Clarity)：**
    捨棄跨平台彩現不可控的 Emojis，全面改用幾何符號（如 `↗`）與結構化前綴詞（如 `交通 /`、`費用 /`）作為視覺錨點。確保在任何裝置上皆能維持絕對的極簡黑白美學與高掃讀性。
*   **斷點磁吸滾動 (CSS Scroll Snap)：**
    上下滑動瀏覽行程時，透過 CSS 強制卡片對齊螢幕中央，完美固定「點（行程節點）」與「線（時間軸）」的視覺交集，避免長文本造成的滾動碎裂感。
*   **跨裝置體驗降級防護 (Degradation Prevention)：**
    捨棄容易與手機瀏覽器「上一頁/下拉更新」衝突的全螢幕盲滑手勢。改採用頂部顯性的「浮動切換器 (Floating Selectors)」與右下角的「時間定位錨點」，確保在手機與滑鼠環境下皆具備極高的操作精準度。
*   **狀態暫存記憶 (State Persistence)：**
    透過 JavaScript 的 `sessionStorage` 紀錄使用者當前選擇的路線。當使用者點擊跳轉至 Google Maps App 並切換回瀏覽器時，網頁不會因重新整理而丟失先前的路線狀態，維持無縫的任務流。

## / 系統架構 (Architecture)

本專案採用純前端靜態架構，無須部署後端資料庫，完美適配 GitHub Pages 的輕量化託管。

*   **UI 容器：** `index.html` (僅包含介面骨架與 CSS 樣式)
*   **資料來源：** `data.json` (獨立的行程資料檔)
*   **動態渲染：** 原生 JavaScript (透過 Fetch API 讀取 JSON 並生成 DOM，無第三方框架依賴)

### 資料與介面分離的優勢
透過將「行程內容」抽離至獨立的 `data.json`，未來的行程更新或替換（例如從東京改為京都計畫），完全不需修改任何 HTML/CSS 程式碼，大幅降低維護成本與改壞畫面的風險。

## / 資料結構指南 (Data Schema)

所有的行程內容皆由 `data.json` 驅動，修改時請嚴格遵守以下的 JSON 結構：

```json
{
  "config": {
    "siteTitle": "網頁標題",
    "dayRoutes": [
      { "id": "a", "label": "路線A名稱" },
      { "id": "b", "label": "路線B名稱" }
    ]
  },
  "schedule": [
    {
      "timeId": "time-10",
      "timeLabel": "10:00",
      "phase": "day",
      "routes": {
        "a": {
          "title": "景點名稱",
          "meta": ["交通 / 資訊", "費用 / 資訊"],
          "description": "景點描述...",
          "links": [
            { "label": "↗ 導航按鈕", "url": "https://..." }
          ]
        }
      }
    }
  ]
}
